# 🧾 Git 규칙 및 협업 전략

본 프로젝트는 원활한 협업과 코드 관리의 일관성을 위해 다음과 같은 Git 컨벤션을 따른다.

---

## 📌 1. Commit Convention

커밋 메시지는 Conventional Commit 스타일을 따른다.

### 커밋 메시지 형식

```
type: 작업 내용 (#이슈번호)
```

### 예시

```
feat: 로그인 기능 구현 (#12)
fix: 회원가입 유효성 검사 오류 수정 (#24)
refactor: 사용자 서비스 로직 분리 (#31)
docs: README 수정 (#8)
```

### Commit Type

| 타입 | 설명 |
|------|------|
| `feat` | 새로운 기능 추가 |
| `fix` | 버그 수정 |
| `docs` | 문서 수정 |
| `style` | 코드 스타일 수정 (공백, 세미콜론 등) |
| `refactor` | 기능 변경 없는 코드 구조 개선 |
| `test` | 테스트 코드 추가 및 수정 |
| `chore` | 빌드 설정, 패키지 관리 등 기타 작업 |

### Commit 작성 규칙

- 타입으로 시작한다
- 타입 뒤에는 `: ` 를 사용한다
- 작업 내용을 간결하게 작성한다
- 관련 Issue가 있다면 `(#이슈번호)`를 포함한다

---

## 🐬 2. Issue Convention

작업은 Issue 기반으로 관리한다.

### Issue 제목 형식

```
type: 기능 설명
```

### 예시

```
feat: 로그인 기능 구현
fix: 회원가입 오류 수정
docs: Git 컨벤션 문서 작성
```

### Issue 작성 구조

```markdown
### 무엇을?

추가하려는 기능 또는 해결하려는 문제를 간단히 설명합니다.

### 작업 상세 내용

- [ ] 작업 내용 1
- [ ] 작업 내용 2
- [ ] 작업 내용 3

### 참고 자료 (선택)

관련 문서나 참고 링크가 있다면 작성합니다.
```

---

## 🚀 3. Pull Request Convention

모든 작업 브랜치는 Pull Request를 통해 `dev` 브랜치에 병합한다.

### PR 제목 형식

PR 제목은 커밋 메시지 형식과 동일하게 작성한다.

```
type: 작업 내용 (#이슈번호)
```

### 예시

```
feat: 로그인 기능 구현 (#12)
fix: 가계부 수정 오류 해결 (#18)
```

### PR 본문 구조

```markdown
## 관련 Issue

closes #이슈번호

## 작업 내용

이번 PR에서 작업한 내용을 간략히 설명합니다.

## 변경 사항

- 변경 내용 1
- 변경 내용 2
- 변경 내용 3

## 스크린샷 (선택)

UI 변경 사항이 있다면 첨부합니다.

## 리뷰 포인트 (선택)

리뷰어가 중점적으로 확인해주었으면 하는 부분을 작성합니다.
```

## 🌿 4. Branch Strategy

본 프로젝트는 `main` / `dev` 브랜치 전략을 사용한다.

### 기본 브랜치

| 브랜치 | 설명 |
|--------|------|
| `main` | 최종 배포용 안정 브랜치 |
| `dev` | 개발 통합 브랜치 |

### 작업 브랜치

모든 기능 개발은 `dev` 브랜치에서 작업 브랜치를 생성하여 진행한다.

**브랜치 형식**

```
type/이슈번호-기능명
```

**예시**

```
feat/12-login
fix/24-signup-error
refactor/31-user-service
docs/7-git-convention
```

### 작업 브랜치 타입

| 타입 | 설명 |
|------|------|
| `feat` | 새로운 기능 개발 |
| `fix` | 버그 수정 |
| `refactor` | 코드 구조 개선 |
| `docs` | 문서 수정 |

---

## 🔀 5. 협업 Workflow

기본 개발 흐름은 다음과 같다.

1. Issue 생성
2. `dev` 브랜치에서 작업 브랜치 생성
3. 기능 개발 및 commit
4. 원격 브랜치 push
5. Pull Request 생성
6. 코드 리뷰 진행
7. 승인 후 merge

---

## 🔒 6. Merge Rule

Merge는 다음 규칙을 따른다.

### Merge 방식

- **Merge Commit** 방식 사용

### Merge 조건

- 최소 2명 이상 승인
- 모든 리뷰 코멘트 확인 후 merge
- 충돌(conflict)이 발생한 경우 PR 작성자가 해결

---

## 🛡️ 7. Branch Protection

협업 안정성을 위해 다음 규칙을 적용한다.

### 보호 브랜치

- `main`
- `dev`

### 규칙

- `main` 브랜치 direct push 금지
- `dev` 브랜치 direct push 금지
- 모든 변경 사항은 Pull Request를 통해 반영
- 승인 없이 merge 금지

---

## ✅ Summary

| 항목 | 규칙 |
|------|------|
| 브랜치 전략 | `main` / `dev` |
| 작업 브랜치 | `type/이슈번호-기능` |
| 커밋 메시지 | `type: 내용 (#이슈번호)` |
| PR 제목 | commit 형식과 동일 |
| Merge 방식 | Merge commit |
| PR 승인 | 2명 이상 |
| direct push | `main` / `dev` 금지 |

## 💡 권장 사항

- 커밋 메시지만 보고도 작업 내용을 이해할 수 있도록 작성한다.
- 기능 단위로 커밋을 나눈다.
- PR은 가능한 작게 유지하여 리뷰 효율을 높인다.
